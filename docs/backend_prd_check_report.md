# 后端功能对照 PRD 完整性检查报告

> 检查范围：`src/DeclarationManagement.Api`（后端 API）
> 
> 结论时间：$(date '+%Y-%m-%d')

## 总体结论

当前后端**已覆盖核心主链路**（登录、用户管理、任务管理、申报、预审/初审、附件、统计导出、归档），但从 PRD 的“是否已经完善”标准看，结论为：

- **核心可用：是**
- **按 PRD 完整交付：否（仍有关键缺口）**

## 已实现（与 PRD 对齐）

1. 认证与用户基础能力
   - JWT 登录、当前用户信息、修改密码已提供。
   - 用户管理支持新增/修改/删除/重置密码。

2. 多选权限模型（符合特别约束）
   - 部门预审权限与初审权限使用独立关联表（非逗号拼接），并在 DbContext 中配置复合主键。

3. 申报任务管理
   - 任务列表、新建、时间窗更新、启停能力已提供。

4. 申报主流程
   - 支持创建、修改、提交、驳回后重提。
   - 提交时校验任务启用状态与时间窗。

5. 审批流关键约束
   - 审核动作会同时写入审核记录（`DeclarationReviewRecord`）和流程日志（`DeclarationFlowLog`）。
   - 审核记录按新增方式累加保留。

6. 统计与导出
   - 提供查询接口。
   - 支持 Excel 导出。
   - 归档仅导出“初审通过”数据并生成 PDF+附件 ZIP。

## 主要缺口（影响“完善”判定）

1. 权限边界未落实到控制器层
   - PRD 定义“统计、用户管理、任务管理”为超管能力，但当前对应控制器仅做 `[Authorize]`，未见超管鉴权限制。

2. 统一错误响应不完整
   - 虽有统一成功响应 `ApiResponse<T>`，但业务中大量 `throw InvalidOperationException`，未见全局异常中间件将异常统一包装为标准失败结构，可能导致错误响应格式不一致。

3. 列表筛选维度与 PRD 不完全一致
   - `mine/pending/statistics` 的查询 DTO 目前主要覆盖时间、部门、类别、状态。
   - PRD 还强调如“项目等级/奖项级别/申报任务/更多列筛选”与类 Excel 的多维筛选，后端查询参数尚未完整体现。

4. 统计 Excel 字段未达到 PRD 明细要求
   - 当前 Excel 列较少，仅基础字段与处理意见。
   - PRD 要求包含“审核部门、原因、认定等级/奖项/金额、备注”等字段，当前未完整导出。

5. 审核记录“查看权限”与详情访问控制偏宽
   - `GetDetailAsync`、附件下载/列表、审核记录查询中，未见按“申报人/当前审核人/超管”的严格可见性校验，存在越权读取风险。

6. “查看审核历史与流程日志”中的流程日志接口缺失
   - 已写入流程日志表，但未提供独立读取流程日志 API。

7. 申报表字段回传不完整
   - `DeclarationDetailDto` 未包含 PRD 表单中的多个文本字段（如认定批文名称、盖章单位及时间、项目内容、项目成果）以及附件集合，需要额外接口拼装，前端表单回显实现成本较高。

## 建议优先级（下一步）

1. **P0**：补齐权限策略（超管策略 + 审核人/申报人数据可见性约束）。
2. **P0**：补齐全局异常处理中间件，确保错误也走统一响应结构。
3. **P1**：扩展筛选 DTO 与查询条件，补齐 PRD 的筛选维度。
4. **P1**：补齐 Excel 导出字段（尤其审核结论相关字段）。
5. **P1**：新增流程日志查询接口，并在详情页可联合展示“审核记录 + 流程日志”。
6. **P2**：优化详情 DTO 聚合，减少前端额外请求和字段拼接复杂度。

## 复核结论

如果你的目标是“可演示主流程”，现在后端已经具备基础能力；
如果你的目标是“严格按 PRD 说已完善”，**目前还不能判定为完善**，建议按上述 P0/P1 缺口补齐后再做一次验收。
